# 优化方案
## 性能优化
1. Q：如果这是一款全校爆款优惠券，几千人几万人同时抢，你这个 Set 会变得巨大无比。Redis 是单线程的，SISMEMBER 复杂度是 O(1) 没问题，但如果你后续需要维护这个 Set（比如回滚库存时要 SREM），或者在大键（BigKey）删除时，会造成 Redis 阻塞。你有没有考虑到 BigKey 问题？对于超高并发的秒杀，Set 真的是最好的选择吗？
A：可以使用分片Set来解决这个问题，比如根据用户ID对一个较小的分片数量取模，将用户分散到多个Set中，这样每个Set的大小就会大大减少，避免了BigKey问题。
如果追求极致，可以使用Redisson框架对Roaring Bitmap的封装RBitSet。


## 可用性优化
### 讨论
1. Q: 场景B:Redis预扣减成功 -> MQ发送失败(比如Broker宕机)，用户前端收到“下单成功”，但后面订单根本没这怎么办?
A: 方案 A：RocketMQ 事务消息
   1. 发送 Half Message（半消息）到 MQ。
   2. 执行 本地事务：执行 Redis Lua 脚本。
   3. 根据 Lua 结果 Commit 或 Rollback 消息。
```java
@Service
public class VoucherOrderService {
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void placeOrder(OrderDTO order) {
        // 1. 创建消息
        Message<OrderDTO> message = MessageBuilder.withPayload(order).build();
        
        // 2. 发送事务消息 (注意：这里不会立即投递给消费者)
        // arg 参数可以传递给本地事务执行器
        rocketMQTemplate.sendMessageInTransaction("voucher_tx_group", "order_topic", message, order);
    }
}

@RocketMQTransactionListener(txProducerGroup = "voucher_tx_group")
public class VoucherTxListener implements RocketMQLocalTransactionListener {

    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private DefaultRedisScript<Long> luaScript; // 你的库存扣减脚本

    /**
     * 第一阶段：执行本地业务（扣减 Redis）
     */
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        OrderDTO order = (OrderDTO) arg;
        try {
            // 执行 Lua 脚本：判断库存、去重、扣减
            Long result = redisTemplate.execute(luaScript, 
                            Arrays.asList("stock:" + order.getVoucherId(), "users:" + order.getVoucherId()), 
                            order.getUserId().toString());

            if (result == 0) {
                return RocketMQLocalTransactionState.COMMIT; // 成功：提交消息，消费者可见
            } else {
                return RocketMQLocalTransactionState.ROLLBACK; // 失败：丢弃消息
            }
        } catch (Exception e) {
            // 发生未知异常（如 Redis 暂时挂了），返回 UNKNOWN
            // 此时 MQ 会进入等待，稍后触发回查逻辑
            return RocketMQLocalTransactionState.UNKNOWN;
        }
    }

    /**
     * 第二阶段：回查逻辑（解决你提到的宕机/丢数据问题）
     * 如果 executeLocalTransaction 返回 UNKNOWN，或者执行中途程序崩溃，MQ Server 会调用此方法
     */
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        OrderDTO order = (OrderDTO) msg.getPayload(); // 实际开发需反序列化
        
        // 【关键】检查 Redis 状态：刚才的 userId 是否已经在 Set 集合里了？
        Boolean isBought = redisTemplate.opsForSet().isMember("users:" + order.getVoucherId(), order.getUserId().toString());
        
        if (Boolean.TRUE.equals(isBought)) {
            return RocketMQLocalTransactionState.COMMIT;
        } else {
            // 这里要小心：可能是 Lua 还没执行，也可能是执行失败
            // 建议配合一个短期的“处理中”状态位，或者直接返回 ROLLBACK
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
}
```

Q: 用 RocketMQ 事务消息：先发 Half 消息，再执行 Lua（扣减 Redis 库存），成功则 Commit。
这听起来保证了“消息发送”和“Redis 扣减”的原子性。但是，消费者端（Consumer） 呢？
A: 在程序中捕获异常，检查是暂时异常还是业务异常，如果是业务异常则执行lua代码的rollback逻辑，redis的库存+1，userid从set中remove，如果不是则让MQ自动重试，如果多次重试后还是失败了，则执行lua代码rollback逻辑

Q:回滚失败了怎么办？
A:

Q：假设使用了 RocketMQ 的重试机制，消费者线程 A 处理消息卡顿了（比如 Full GC），超时未提交，RocketMQ 会重试投递给另一个消费者线程 B。此时，如果 A 恢复了，B 也可能在处理同一条消息，这就导致了重复消费问题。你怎么解决这个问题？
A：可以为该消息设置一个唯一的 ID（比如订单 ID），在消费者端使用分布式锁（如 Redis 分布式锁）来确保同一时间只有一个线程在处理该消息。处理前尝试获取锁，成功则继续处理，失败则说明另一个线程已经在处理该消息，可以直接丢弃或等待锁释放后再重试。

### 总结
可见以上的优化方案在不断的追问下复杂度逐步提高，从最初的简单的发送消息到MQ，到使用RocketMQ的事务消息，再到处理消费者端的重复消费问题，最后还涉及到分布式锁的使用。这些优化方案虽然可以提高系统的可靠性和一致性，但也增加了系统的复杂度和维护成本。
最后，在同学的建议下，选择了复杂度更低，但是也是高可用的方案：
1. 入口：Lua 脚本原子预扣减（解决超卖、排队）。
2. 中转：普通消息 + 本地表补偿（解决消息丢失）。
3. 落库：数据库事务 + 唯一索引（解决幂等、持久化）。
4. 兜底：延时队列（解决超时未支付问题）。

    // 1. 执行 Lua 脚本进行 Redis 预扣减
    Long result = redisTemplate.execute(luaScript, keys, order.getUserId());
    
    if (result != 0) {
        throw new BusinessException("库存不足或重复购买");
    }

    // 2. 尝试发送普通 MQ 消息
    try {
        // 使用同步发送，确保能捕获到发送失败的异常
        SendResult sendResult = mqProducer.send("order_topic", JSON.toJSONString(order));
        
        if (!sendResult.isSuccess()) {
             throw new Exception("MQ服务器返回发送失败");
        }
    } catch (Exception e) {
        // 3. 【核心逻辑】发送失败，记录到本地异常表
        log.error("MQ发送失败，进行本地记录兜底: {}", order, e);
        saveToErrorLog(order, e.getMessage());
    }
}

private void saveToErrorLog(OrderDTO order, String reason) {
    MqErrorLog errorLog = new MqErrorLog();
    errorLog.setMessageBody(JSON.toJSONString(order));
    errorLog.setTopic("order_topic");
    errorLog.setRetryCount(0);
    errorLog.setStatus(0);
    // 存储到 MySQL
    mqErrorLogMapper.insert(errorLog);
}

@Scheduled(fixedRate = 60000)
public void retryFailedMessages() {
    // 1. 查询状态为 0 且重试次数 < 5 的记录
    List<MqErrorLog> logs = mqErrorLogMapper.selectPendingLogs();

    for (MqErrorLog errorLog : logs) {
        try {
            // 2. 重新发送消息
            mqProducer.send(errorLog.getTopic(), errorLog.getMessageBody());
            
            // 3. 发送成功，更新状态为 1
            errorLog.setStatus(1);
            mqErrorLogMapper.updateById(errorLog);
        } catch (Exception e) {
            // 4. 发送失败，重试次数 +1
            errorLog.setRetryCount(errorLog.getRetryCount() + 1);
            mqErrorLogMapper.updateById(errorLog);
            
            if (errorLog.getRetryCount() >= 5) {
                // 钉钉/邮件告警，人工介入
                alertService.send("消息补偿多次失败，请检查MQ集群状态！");
            }
        }
    }
}

@Component
@RocketMQMessageListener(topic = "order_topic", consumerGroup = "voucher_consumer_group")
public class OrderConsumer implements RocketMQListener<String> {

    @Autowired
    private IVoucherOrderService voucherOrderService;

    @Override
    public void onMessage(String message) {
        // 1. 解析消息
        OrderDTO orderDTO = JSON.parseObject(message, OrderDTO.class);

        try {
            // 2. 执行数据库操作
            voucherOrderService.createVoucherOrder(orderDTO);
        } catch (DuplicateKeyException e) {
            // 3. 幂等处理：如果是数据库唯一索引报错，说明消息重复投递了
            log.warn("订单已存在，无需重复创建: {}", orderDTO.getOrderId());
        } catch (Exception e) {
            // 4. 其他异常：抛出异常让 MQ 重试
            log.error("创建订单失败，触发MQ重试: {}", message, e);
            throw e; 
        }
    }
}

@Service
public class VoucherOrderServiceImpl implements IVoucherOrderService {

    @Transactional
    public void createVoucherOrder(OrderDTO orderDTO) {
        // 1. 扣减数据库库存 (带上 stock > 0 预防超卖)
        // SQL: update tb_voucher_stock set stock = stock - 1 where voucher_id = ? and stock > 0
        boolean success = voucherService.updateStock(orderDTO.getVoucherId());
        
        if (!success) {
            // 理论上 Lua 已经预检过了，如果这里失败，说明 DB 和 Redis 数据不一致，需记录异常
            throw new RuntimeException("数据库库存扣减失败");
        }

        // 2. 创建订单记录
        // 数据库 voucher_order 表必须有唯一索引：voucher_id + user_id
        VoucherOrder order = new VoucherOrder();
        order.setId(orderDTO.getOrderId()); // 使用发过来的 orderId
        order.setUserId(orderDTO.getUserId());
        order.setVoucherId(orderDTO.getVoucherId());
        order.setStatus(1); // 1: 待支付
        save(order);

        // 3. 发送延时消息（用于超时关单）
        // 投递到延时队列，比如 30 分钟后检查
        sendDelayMessage(orderDTO);
    }
}

@Component
public class DelayOrderConsumer {

    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 延时消息监听
    public void onMessage(OrderDTO order) {
        // 1. 尝试关单
        // SQL: update tb_order set status = 5 where id = ? and status = 1
        boolean isClosed = orderService.closeOrder(order.getOrderId());

        // 2. 如果关单成功，或者订单已经是“超时取消”状态（应对重试场景）
        if (isClosed || orderService.checkStatus(order.getOrderId(), 5)) {
            // 注意：Redis 回滚脚本必须实现幂等性（SREM 若已经是 0 则不处理 stock）
            executeRedisRollback(order);
            log.info("订单超时关单，Redis 库存回滚完成: {}", order.getOrderId());
        } else {
            log.info("订单已支付或人工取消，无需回滚: {}", order.getOrderId());
        }
    }

    private void executeRedisRollback(OrderDTO order) {
        // 计算分片 Set 的 Key 
        String setKey = "voucher:users:" + order.getVoucherId() + ":" + (order.getUserId() % 100);
        String stockKey = "voucher:stock:" + order.getVoucherId();

        redisTemplate.execute(rollbackLuaScript, 
            Arrays.asList(stockKey, setKey), 
            order.getVoucherId().toString(), 
            order.getUserId().toString());
    }
}

## 总结
### 安全
gateway 筛选入口，进行鉴权，鉴权通过后将用户信息转化为 JWT 传给下游微服务。由于引入 mTLS 的复杂度较高，选择对称加密生成 JWT 的方式来保证请求的合法性，防止伪造请求和非法横向调用。（无法防止内部爆破场景）

服务间调用时，使用 Feign 的 RequestInterceptor 将 JWT 继续透传下去，确保每一层服务都能验证请求的合法性。如果担心异步线程调用其他服务，可以使用阿里开源的 TransmittableThreadLocal 来实现父线程内容到子线程的拷贝，确保异步调用也能正确传递用户信息。（TTL 是对 ITL 的改进，ITL 只能创建线程的同时将 ThreadLocal 内容拷贝到新线程中，不能拷贝到线程池的线程）

### OSS
Temp 目录 + Lifecycle + 业务触发 Copy/Move：

1. 前端只允许直传到临时目录 (temp/)：后端给前端下发的预签名 URL，其 key（路径）被硬编码限制必须是以 temp/yyyy-MM-dd/<uuid>.jpg 开头的。如果前端尝试修改路径上传到正式目录（比如 posts/），OSS 会校验签名失败并拒绝。
2. 配置 OSS 生命周期规则： 在云厂商的后台控制台，直接配置一条规则：“任何在 temp/ 目录下的文件，只要创建时间超过 24 小时（或者 1 小时），自动删除”。这一步不需要后端写任何定时任务代码，完全是云厂商底层的自动清理，既没有内存溢出风险，也不会漏删。
3. 业务落库与状态反转（前端通知后端）：
- 前端上传到 temp/ 成功后，调用后端的“发布文章”接口，把这个 temp/ 下的 URL 传给后端。
- 后端在落库文章数据的同时（可以在同一个事务中，或者通过本地消息表/MQ 保证最终一致性），调用 OSS 的内部 SDK，执行一个 CopyObject 操作（在云端内部非常快），把文件从 temp/... 复制到正式目录 posts/yyyy-MM/uid/...。
- 最终指向真实目录： 后端数据库里存的，永远是 posts/ 开头的正式地址。前端也是展示这个地址。

### MCP
将MCP改造为使用带有服务发现机制的网络通信（像 gRPC 或基于云原生的 SSE/WebSockets 版本的 MCP），来达到解耦的目的。
