## 关键词搜索

### TF-IDF 与 BM25
TF-IDF和BM25是两种常用的文本检索算法，此处不过多论述它们的原理，只做对比，阐述为什么BM25更适合于搜索引擎。

#### 词频饱和度
- TF-IDF： 词频越高，得分越高，且几乎是线性增长的。如果一个词在文档里出现了 100 次，它的权重会比出现 10 次的高得多。
- BM25： 引入了“饱和”机制。它认为：一个词出现 1 次和 0 次区别很大，出现 10 次和 1 次区别也很大，但出现 100 次和 1000 次的区别其实很小。

场景影响：在 TF-IDF 下，一篇疯狂重复关键词的“水贴”可能得分极高；而在 BM25 下，当词频达到一定程度（由参数 $k_1$ 控制）后，得分就会趋于平缓，不再无限制增加。

#### 文档长度处理
- TF-IDF： 虽然也有简单的长度惩罚，但效果不够精细。长文章天然包含更多词汇，更容易在 TF-IDF 中获得高分。
- BM25： 引入了平均文档长度（avgdl）进行对比。如果一篇文档的长度远大于平均长度，那么它的得分会被适当降低；反之，如果文档较短，得分可能会得到提升。

#### 公式对比
- TF-IDF： $score(d, q) = \sum_{t \in q} tf(t, d) \cdot idf(t)$
- BM25： $score(d, q) = \sum_{t \in q} idf(t) \cdot \frac{tf(t, d) \cdot (k_1 + 1)}{tf(t, d) + k_1 \cdot (1 - b + b \cdot \frac{|d|}{avgdl})}$
其中，$k_1$ 控制词频饱和度，$b$ 控制文档长度的影响。

在实际应用中，BM25 提供了可变参数 $k_1$ 和 $b$，这两个参数影响的就是以上所说的两点区别。

### 稀疏向量
我们先介绍一下稀疏向量的产生过程：
1. 分词： 将原始文本分解为 Token。
2. 词表映射： 稀疏向量的维度通常对应于整个预训练模型的 词表大小（通常是 30,000 维以上）。
3. 权重预测： 模型会为文本中的每个词（甚至包括没出现但相关的词）计算一个权重分。

形式： 生成的结果是一个类似键值对的结构：{3045: 1.5, 12021: 2.8, ...}。其中数字是<span class="term-tooltip" data-tooltip="稀疏向量的每一个维度都直接对应词表里的一个具体词汇" tabindex="0">词表中的索引</span>，浮点数是<span class="term-tooltip" data-tooltip="该维度的分值代表了该词对于这篇文档的 区分度。分值越高，说明这个词越能代表这篇文档的核心内容。" tabindex="0">权重</span>。由于大多数维度的值为 0，只有少数与文本相关的维度有值，因此被称为“稀疏”。

稀疏向量（Token -> 权重值）区别于传统的倒排索引（Token -> 文档列表）模型的点在于**同义词扩展**：这是稀疏向量最强大的地方。例如原始文本是“西红柿炒蛋”，传统的 BM25 在“番茄”维度上是 0。但 SPLADE 等模型能识别出语义联系，会在“番茄”这个维度也给出一个较小的权重。

### 在milvus应用
在 Milvus 中，稀疏向量的生成通常是通过预训练模型（如 SPLADE）对文本进行编码得到的。生成的稀疏向量可以直接存储在 Milvus 中，并用于后续的搜索和检索。
```python
from pymilvus import (
    MilvusClient, DataType, Function, FunctionType
)
# 创建 Milvus 客户端
client = MilvusClient(
    uri="http://localhost:19530",
    token="root:Milvus"
)

# 定义数据表结构
schema = client.create_schema(auto_id=False)
schema.add_field(field_name="id", datatype=DataType.INT64, is_primary=True, description="product id")
schema.add_field(field_name="text", datatype=DataType.VARCHAR, max_length=1000, enable_analyzer=True, description="raw text of product description")
schema.add_field(field_name="text_dense", datatype=DataType.FLOAT_VECTOR, dim=768, description="text dense embedding")
schema.add_field(field_name="text_sparse", datatype=DataType.SPARSE_FLOAT_VECTOR, description="text sparse embedding auto-generated by the built-in BM25 function")
# 图像的密集向量（根据数据集内容可选）
schema.add_field(field_name="image_dense", datatype=DataType.FLOAT_VECTOR, dim=512, description="image dense embedding")
# 定义BM25函数
bm25_function = Function(
    name="text_bm25_emb",
    input_field_names=["text"], # 原始文本字段输入
    output_field_names=["text_sparse"], # 稀疏向量字段输出
    function_type=FunctionType.BM25, # 内置BM25函数类型
)
schema.add_function(bm25_function)
```

## 语义搜索
语义搜索是随着大模型的发展而兴起的一种搜索方式，它不再仅仅依赖于关键词的匹配，而是通过理解文本的语义来进行搜索。

### 密集向量
语义搜索的核心是密集向量。与稀疏向量不同，密集向量通常是一个<span class="term-tooltip" data-tooltip="例如在milvus中用于存储文本的密集向量维数为768， 用于存储产品图像的密集向量维数为512" tabindex="0">固定长度的实数数组</span>，每个维度都具有一定的数值。这个向量是通过预训练模型对文本进行编码得到的。

#### 分片
在实践中，我需要对[文档](../../../../pdf/8.深小友-软件使用说明.pdf)进行分片处理。可以看到该文档的特征有几个：
1. 分层结构明显，且各层级文字少，图片较多
2. 每层的内容较为独立

根据以上特征，我选择了**将图片解释为文本语言后基于层级结构分片的方法**，以上方法可以很好的满足用户对功能的“是什么，怎么做”以及“图片里面有什么”的需求。

*tips：如果希望用户能够直接通过图片进行搜索，那么就需要将图片的特征向量也存储在 Milvus 中，这样用户就可以通过上传图片来进行搜索了。*

此外，由于各个层级的内容较为独立，分片之间并不需要太多的重叠度，因此在实践中没有设置重叠，转而使用路径来标识不同的分片。
例如：[路径：软件简介 > 首页功能 > 帖子详情]2.4.3.2 点赞帖子：用户可以通过点击“点赞”按钮...

以下是代码简要逻辑：
```python
class ShenXiaoyouProParser:
    def __init__(self, docx_path, api_key, output_dir="rag_data"):
        self.docx_path = docx_path
        self.output_dir = output_dir
        self.img_dir = os.path.join(output_dir, "images")
        self.client = ZhipuAI(api_key=api_key)
        
        if not os.path.exists(self.img_dir):
            os.makedirs(self.img_dir)
            
        self.doc = Document(docx_path)
        # 预加载文档中所有的图片关系映射
        self.image_blobs = self._get_image_blobs()

    def _get_image_blobs(self):
        """建立 rId 到图片二进制数据的映射"""
        images = {}
        for r in self.doc.part.rels.values():
            if r.reltype == RT.IMAGE:
                images[r.rId] = r.target_part.blob
        return images

    def _get_glm_analysis(self, image_bytes):
        """调用 GLM-4V 真实接口"""
        img_base64 = base64.b64encode(image_bytes).decode('utf-8')
        try:
            response = self.client.chat.completions.create(
                model="glm-4v",
                messages=[
                    {
                        "role": "user",
                        "content": [
                            {"type": "text", "text": "描述此小程序UI截图：1.页面功能名称 2.核心按钮/文字 3.展示的状态逻辑。用于RAG检索，请用中文简短回答。"},
                            {"type": "image_url", "image_url": {"url": img_base64}}
                        ]
                    }
                ]
            )
            return response.choices[0].message.content
        except Exception as e:
            print(f"GLM API 调用失败: {e}")
            return "图片解析异常"

    def run(self):
        chunks = []
        current_path = []
        current_chunk = None

        # 遍历文档段落
        for para in self.doc.paragraphs:
            text = para.text.strip()
            
            # 匹配层级标题 (如 2.4 或 2.4.1)
            header_match = re.match(r'^(\d+(?:\.\d+)*)\s+(.*)', text)
            if header_match:
                if current_chunk: chunks.append(current_chunk)
                
                num_part = header_match.group(1)
                title_part = header_match.group(2)
                level = num_part.count('.')
                
                # 动态维护层级路径
                if len(current_path) <= level:
                    current_path.extend([''] * (level - len(current_path) + 1))
                current_path[level] = title_part
                current_path = current_path[:level + 1]
                
                current_chunk = {
                    "metadata": {
                        "id": str(uuid.uuid4()),
                        "path": " > ".join(current_path),
                        "header": text
                    },
                    "content": "",
                    "images": [],
                    "vector_text": f"【{' > '.join(current_path)}】"
                }
                continue

            # 处理正文与图片
            if current_chunk:
                if text:
                    current_chunk["content"] += text + "\n"
                    current_chunk["vector_text"] += text + " "

                # 提取段落中的图片
                # 查找 XML 中的 blip 标签以获取 rId
                xml_str = para._p.xml
                r_ids = re.findall(r'r:embed="([\w\d]+)"', xml_str)
                
                for rid in r_ids:
                    if rid in self.image_blobs:
                        img_data = self.image_blobs[rid]
                        img_filename = f"{uuid.uuid4().hex}.png"
                        img_path = os.path.join(self.img_dir, img_filename)
                        
                        # 持久化图片
                        with open(img_path, "wb") as f:
                            f.write(img_data)
                        
                        # 获取 GLM 描述
                        print(f"正在通过 GLM 解析图片: {img_filename}...")
                        description = self._get_glm_analysis(img_data)
                        
                        current_chunk["images"].append({
                            "local_path": img_path,
                            "ai_description": description
                        })
                        # 核心优化：将视觉描述融入向量文本
                        current_chunk["vector_text"] += f" [描述: {description}]"

        if current_chunk:
          # 只有当内容长度超过一定阈值，或者该标题下确实有图片时，才视为有效分片
          if len(current_chunk["content"].strip()) > 10 or current_chunk["image_assets"]:
              chunks.append(current_chunk)
          else:
              # 如果是空分片（只有标题），我们可以记录下这个父级标题
              # 确保下一个子分片的路径能继承它
              last_empty_header = current_chunk["metadata"]["header"]
        
        # 结果持久化
        output_file = os.path.join(self.output_dir, "processed_rag_data.json")
        with open(output_file, 'w', encoding='utf-8') as f:
            json.dump(chunks, f, ensure_ascii=False, indent=2)
        
        print(f"处理完成！共生成 {len(chunks)} 个分片。数据保存在: {output_file}")
        return chunks

# --- 执行入口 ---
if __name__ == "__main__":
    # 请替换为您真实的智谱 API Key
    MY_API_KEY = "your_zhipu_api_key_here"
    DOC_PATH = "8. 深小友-软件使用说明.doc" # 请确保文件已转换为 .docx 格式
    
    parser = ShenXiaoyouProParser(DOC_PATH, MY_API_KEY)
    final_result = parser.run()
```

#### 召回
召回方案分为两个阶段：粗排阶段的“倒数排名融合（RRF）”和精排阶段的“模型重排序（Rerank）”

##### 第一阶段：RRF
RRF 是目前 Milvus 和 Elasticsearch 处理多路召回最常用的算法。它不看分数的绝对值，只看排名。

计算公式：$$Score = \sum_{d \in D} \frac{1}{k + rank_i(d)}$$
- $rank_i(d)$：文档 $d$ 在第 $i$ 路检索（如 BM25）中的排名次序。
- $k$：一个平滑常数（通常设为 60），防止排名靠前的文档权重过大。

如果一个分片在 BM25 里排第 1，在向量检索里排第 2，它的 RRF 分数就会非常高。这种方式天然地平衡了两者的优势。

##### 第二阶段：Rerank
RRF 虽然解决了分数对齐问题，但它依然是“盲目”的，因为它不理解用户问题与分片的深度逻辑。为了达到最高的召回精度，必须引入 Rerank 模型。

1. 运作逻辑
  1. 从 RRF 合并后的结果中取出 Top 50（粗排结果）。
  2. 将“用户问题”和“分片内容”两两配对，喂给 Rerank 模型。
  3. Cross-Attention 机制：不同于向量检索只做特征计算，Rerank 模型会将问题和文档放在一起进行深度交叉计算，输出一个 0-1 之间的相关性概率分。

***为什么 Rerank 更准？***
比如《深小友》手册中，“2.6.4 新的朋友”和“2.6.5 新的评论”，在向量空间可能离得很近。但 Reranker 模型能精准识别出“评论”和“朋友”这两个字在语义逻辑上的巨大区别。除此之外，向量模型会压缩信息导致细节丢失，而 Reranker 能逐字逐句对比相似度。
